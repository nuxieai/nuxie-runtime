# Binary Import Prototype

Ticket: `#5`

Scope lock: binary import completion is now governed by
[`binary-import-completion-contract.md`](binary-import-completion-contract.md).
Use that document to decide whether new parity work belongs in `nuxie-binary` or
should move to `nuxie-graph`/later runtime crates.

The current public surface classification is recorded in
[`binary-import-surface-audit.md`](binary-import-surface-audit.md).

Question: can Rust decode runtime `.riv` files into a stable object arena while preserving object order, type keys, property-key dispatch, inherited properties, and forward-compatible skipping?

Prototype command:

```sh
make inspect
```

The command imports `fixtures/graph/dependency_test.riv` and prints JSON with header data, object indexes, type keys, type names, decoded properties, skipped non-stored properties, and C++ import-stack keep/drop status.

The library path is `crates/nuxie-binary`; the inspect CLI is `crates/nuxie-binary/src/bin/riv-inspect.rs`.

Verdict: the binary import shape works for the starter corpus.

Implemented:

- Runtime header parsing with `RIVE` fingerprint, version, file ID, and property ToC decoding.
- Header integer and malformed-header semantics match the C++ runtime: same-major newer minor versions are accepted, major-version mismatches are rejected as `unsupportedVersion`, bad/truncated fingerprints and incomplete required header segments are malformed, noncanonical and overwide LEB128 integer encodings are accepted, major/minor/file-id/property-ToC values outside signed C++ `int` range are malformed, boundary ToC keys at `i32::MAX` are accepted, duplicate ToC property keys overwrite earlier field ids with later field ids, truncated packed ToC field-id words and missing property-ToC terminators are malformed at the import boundary, and ToC property keys are preserved as 32-bit values rather than prematurely narrowed to runtime object property-key width.
- Object type keys are read with C++ `int` semantics. Unknown object type keys above Rust's generated `u16` schema key space, including the boundary `i32::MAX` key, are accepted and skipped as null object slots when their properties can be skipped, while values outside signed `int` range are malformed. Object property keys are read with C++ `uint16_t` range semantics; the boundary `u16::MAX` key is accepted and skippable, while values above `u16::MAX` are malformed.
- Runtime `uint` field values match C++ `CoreUintType` range semantics: `u32::MAX` is accepted, and larger varuint payloads are malformed for both normal property decoding and unknown-property skipping.
- LEB128 varuint decoding follows C++ `decode_uint_leb`: noncanonical encodings are accepted, including overwide terminated encodings that rely on the C++ reader's wrapping shift behavior and length prefixes for string/bytes payloads, while truncated varuints in the header, object type key, property key, and string/bytes length positions are malformed, matching C++ reader overflow behavior.
- Runtime primitives: uint, string, bytes, float32/double, color, bool, and callback markers. String values preserve their raw bytes and expose a UTF-8 view only when the payload is valid UTF-8, matching C++ `std::string` byte semantics without forcing lossy conversion. Bool payloads match C++ `CoreBoolType` exactly: only byte `1` is true, and other present byte values decode to false rather than becoming malformed. Bytes values preserve the full raw payload while still exposing length/preview metadata for inspection. Encoded `List<Id>` bytes payloads also expose decoded `uint32_t` ID buffers through `RuntimeObject::id_list_property`, using the same embedded varuint reader semantics as C++ decoders such as `DataBindPath::decodePath`, `DataBindContext::decodeSourcePathIds`, and `DataConverterOperationViewModel::decodeSourcePathIds`; C++ probe comparisons now also verify that cloned `DataConverterOperationViewModel` instances attached directly to data binds and through `DataConverterGroupItem` retain those decoded source path IDs. `Mesh.triangleIndexBytes` exposes a separate decoded `uint16_t` index-buffer view through `RuntimeObject::mesh_triangle_indices`, matching C++ `Mesh::decodeTriangleIndexBytes`, including local range-error-to-zero behavior. File assets expose the C++ `cdnUuidStr()` view through `RuntimeObject::file_asset_cdn_uuid_string`, preserving the raw `cdnUuid` bytes while returning the byte-reordered UUID string only for 16-byte payloads. They also expose C++ `fileExtension()`, `uniqueName()`, and `uniqueFilename()` equivalents, including raw-byte variants that preserve `Asset.name` byte semantics while UTF-8 variants return `None` for invalid UTF-8 output. `RuntimeFile::manifest()` follows the C++ `ManifestAsset` + `FileAssetContents` importer relationship and decodes manifest names and path ID lists for later data-resolver use while keeping the raw content object inspectable, including C++ signed-`int` manifest keys, wrapped high-bit IDs such as `u32::MAX -> -1`, and partial map mutations when malformed manifest bytes make `ManifestAsset::decode()` return false during otherwise successful file-asset resolution. Typed scalar/string helpers expose generated C++ stored member defaults for omitted properties using the schema's C++-verified initializer metadata, including both the UTF-8 and raw-byte views of default strings, while keeping the serialized property list sparse. Object-specific constructor defaults that differ from generated base members are modeled where proven by C++ probes, including `Artboard` inheriting `LayoutComponent.clip` but defaulting it to `true`; a C++ constructor source audit now locks that override surface to the known case. Truncated bytes payloads and truncated fixed-width primitive payloads such as float32/double, color, and bool are malformed when the declared/read width outruns the stream.
- A C++ source audit now also pins the FileAsset helper bodies mirrored by `nuxie-binary`: `uniqueName()`, `uniqueFilename()`, raw `cdnUuid` copying, `cdnUuidStr()` byte ordering/hyphen formatting, and the concrete asset `fileExtension()` override table.
- Object stream parsing into `Vec<Option<RuntimeObject>>` with stable object IDs. The arena intentionally preserves decoded known objects even when C++ would later delete them after `object->import(importStack)` returns `MissingObject`; a parallel `import_statuses` vector records whether each slot is a null object, a successfully imported object, or a dropped object.
- Import-status tracking now uses a small C++ `ImportStack` model with explicit latest-importer keys and `lastAdded` order instead of one boolean per context. This pins keyed ownership for backboard/artboard/file-asset/animation/state-machine/listener/view-model/data-converter/scripted-object importers and preserves C++ `readNullObject()` ordering, including the case where a later `StateMachineLayerImporter` consumes a null object before an older `StateMachineImporter` can turn it into a null input placeholder, and the case where a non-consuming latest importer is skipped while searching backward for a null-object consumer.
- Type-key lookup through `nuxie-schema`.
- Inherited property lookup, so `Node`, `Shape`, `Artboard`, etc. decode `Component.name`, `Component.parentId`, and other ancestor fields. Known-object deserialization dispatch uses primary property keys only, matching the generated C++ `deserialize` switches; alternate keys such as `Node.xArtboard` fall through to the global fallback path and are skipped rather than assigned to the object. Schema-known properties that are not present in the generated C++ `deserialize` switch also route through that same fallback path instead of being decoded directly. Duplicate serialized stored properties overwrite the previous arena value by property key, matching C++ generated deserializers assigning into a single member where the last serialized value wins.
- Import-time C++ mutations that affect decoded property values are mirrored when proven by probes. `BackboardImporter::addFileAsset` rewrites duplicate `FileAsset.assetId` values while assets are added, so Rust normalizes duplicate file-asset IDs in serialized order after object import. The normalization now only considers objects whose `import_statuses` entry is `Imported`, matching C++ deleting contextless `FileAsset` objects before the backboard importer can mutate them.
