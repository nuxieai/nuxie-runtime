Question: what is the smallest useful Rust implementation of the artboard runtime graph lifecycle?

Prototype command:

```sh
make graph
```

The command imports `fixtures/graph/dependency_test.riv`, projects the flat runtime object stream into C++-style file and artboard-owned collections, resolves `Component.parentId` references, builds child indexes, derives explicit dependency edges plus a topological dependency order, and exposes the first renderer-adjacent graph relationships without producing draw commands.

Lifecycle mapping:

- `import`: find serialized `Artboard` ranges, then compact them into the same shape as C++ `Artboard::objects()`: include component objects, artboard-owned user-input/interpolator objects, and null/abstract slots; exclude concrete non-components routed to other runtime lists.
- `import`: project C++-owned collections for `File::assets()`, file view models, data enums, `Artboard::animation(i)`, and `Artboard::stateMachine(i)` from the decoded object stream. File-level assets, view models, and data enums are projected from `rive-binary`'s public `RuntimeFile` collection helpers so graph does not duplicate import-stack ownership rules.
- `on_added_dirty`: resolve local `parentId` values against the artboard-local component index.
- `on_added_dirty`: resolve nullable draw target, draw rule, and clipping source references while preserving the raw serialized IDs for missing-target diagnostics.
- `on_added_dirty`: mirror the C++ initialization quirk that creates an `"Auto Generated State Machine"` when an artboard has neither animations nor explicit state machines.
- `on_added_dirty`: build C++ `children()` lists in object-processing order, including inherited `LayoutComponent::onAddedDirty` adoption of resolved `LayoutComponentStyle` objects, without treating those adopted style children as serialized parent-pointer rewrites.
- `on_added_dirty`: project C++ `TransformComponent::constraints()` lists on
  component nodes from exact `Constraint::onAddedDirty` registrations, without
  running constraint solving, transform dirt propagation, or component updates.
- `on_added_clean`: build clipping relationships for source shape descendants and clipped drawable descendants from the projected child lists.
- `on_added_clean`: project C++ shape-paint container registrations through `ArtboardGraph::shape_paint_containers`, including registered paints, mutators, gradient stops, feathers, stroke effects, and target-effect group links, without running paint mutation, effect execution, gradient stop sorting, path-effect application, renderer paint allocation, draw commands, or GPU work.
- `on_added_dirty`: project C++ `NSlicerDetails` registrations through `ArtboardGraph::n_slicer_details`, including ordered X/Y axes and patch-indexed tile modes for exact `NSlicer` and `NSlicedNode` details owners, without running NSlicer deformation math, patch solving, layout updates, path deformation, or render-path mutation.
- `on_added_clean`: project `Shape::onAddedClean` render-path deformer caches as `shape_deformers`, walking parent ancestors and recognizing the first exact `NSlicedNode` accepted by `RenderPathDeformer::from` without running `NSlicer` deformation math, `Path::buildPath`, gradient deformer updates, or point deformation.
- `on_added_clean`: project C++ skeletal registration caches as `skeletal_bones` and `skeletal_skins`, recording exact `Bone` child lists, IK peer constraints on ancestor bones, exact `Mesh`/`PointsPath` skinnable parents, and valid tendon-to-bone registrations without running bone transforms, IK solving, skin matrix updates, vertex deformation, or dirt propagation.
- `on_added_clean`: project C++ artboard host registries for exact `NestedArtboard`, `NestedArtboardLeaf`, `NestedArtboardLayout`, and `ArtboardComponentList` objects, exposing `nested_artboards`, `component_lists`, and the combined `artboard_hosts` list without running nested-artboard update, component-list layout, cloning, or advance behavior.
- `on_added_dirty`: project C++ `ArtboardListMapRule` registration tables on `ComponentListNode::map_rules`, matching `ArtboardComponentList::addMapRule` overwrite-by-view-model semantics without creating list artboard instances, binding data contexts, sorting draw indices, virtualizing list items, or evaluating map-rule selection.
- `on_added_clean`: project C++ joystick registration facts, exposing `joysticks`, `joysticks_apply_before_update`, resolved custom handle sources, resolved x/y animations, and nested remap dependents collected from keyed animation targets without running `Joystick::apply`, component updates, data-bind scheduling, or animation advancement.
- `on_added_clean`: project C++ reset and advance lifecycle registries, exposing `resetting_components` and `advancing_components` from the exact `ResettingComponent::from` and `AdvancingComponent::from` switches without running `reset()`, `advanceComponent()`, data-bind advancement, component updates, or frame scheduling.
- `import`/`initialize`: project C++ artboard-owned and state-machine-owned data-bind registrations through `ArtboardGraph::data_binds` and `StateMachineGraph::data_binds`; artboard-owned binds include resolved target/converter facts and initialized `sortDataBinds()` order, while state-machine-owned binds use the verified authored state-machine graph ownership from `rive-binary`, without running data-context binding, dirty queues, property observers, converter execution, source/target mutation, state-machine execution, or data-bind advancement.
- `import`: project C++ state-machine scripted-object registrations through `StateMachineGraph::scripted_objects`, including ordered script inputs from `ScriptedObjectImporter::addInput`, without running script asset initialization, VM registration, cloning, script input hydration, script execution, or state-machine execution.
- `build_dependencies`: project C++'s per-`Shape` synthetic `PathComposer` node as a graph fact, with its owning shape and registered paths sourced from `rive-binary`'s imported shape list. The synthetic node participates in `dependency_nodes`, `dependency_node_edges`, and `dependency_node_order`; the compatibility `dependency_order` field remains a filtered real-component local-ID order.
- `on_added_dirty`: project C++ mesh/path vertex registrations through `ArtboardGraph::meshes` and `ArtboardGraph::paths`, including ordered `MeshVertex`/`PathVertex` children and each vertex's resolved `Weight`/`CubicWeight` attachment, without running vertex deformation, skinning math, `Path::buildPath`, contour/path tessellation, weight blending, or dirty propagation.
- `build_dependencies`: derive C++ parent dependency, targeted-constraint, IK-target, IK chain child, draw-target, draw-rule, clipping-source, skinned `Mesh`/`PointsPath`, tendon-bone, skin peer-constraint-parent, Joystick custom-handle, scroll-constraint-to-scroll-bar, scroll-constraint-to-layout-child, path-composer, clipping-shape path-composer, follow-path, text-follow-path, text variation helper, stroke/fill/feather path-builder, effect-parent, and linear/radial gradient paint-container dependency-node edges, order dependency nodes over the explicit edge list, expose a real-component filtered order, and surface dependency-cycle diagnostics. `ComponentNode::graph_order` separately mirrors C++ `Artboard::sortDependencies` by sorting only root-reachable `buildDependencies` edges, so draw-target, draw-rule, and clipping-source projection edges do not affect dirt-depth graph order. C++ parent dependency edges are emitted only when an audited `buildDependencies()` implementation calls `parent()->addDependent(this)`, not for every structural parent/child relation. C++ override exceptions are modeled where audited: `Skin::buildDependencies`, `TargetedConstraint::buildDependencies`, `Joystick::buildDependencies`, `ClippingShape::buildDependencies`, `FollowPathConstraint::buildDependencies`, `TextFollowPathModifier::buildDependencies`, `Stroke::buildDependencies`, `Fill::buildDependencies`, `Feather::buildDependencies`, and `LinearGradient::buildDependencies` do not call their parent dependency implementation, while `IKConstraint::buildDependencies` reuses the targeted edge and adds its own target-to-constraint edge. `Mesh::buildDependencies` also emits its exact parent dependency even though `Mesh` is not a `TransformComponent`. `GroupEffect::buildDependencies` and `ScriptedPathEffect::buildDependencies` preserve inherited dependencies and add explicit effect-parent dependencies.
- `build_dependencies`: project the artboard-local portion of C++
  `Component::dependents()` on `ComponentNode::dependent_locals`, preserving
  `DependencyHelper::addDependent` duplicate suppression. Synthetic
  `PathComposer` and `TextVariationHelper` nodes stay in
  `dependency_nodes`/`dependency_node_edges`, while temporary draw-target roots
  stay out of component local lists.
- `build_dependencies`: project exact C++ list-constraint registration facts by recording `ListFollowPathConstraint` children whose parent is an `ArtboardComponentList`, matching `ConstrainableList::addListConstraint` without running `constrainList()`, list layout, or virtualization.
- `build_dependencies`: project exact C++ layout-constraint registration facts by recording `ScrollConstraint` registrations on exact `LayoutNodeProvider` content children through `layout_constraint_registrations`, matching `LayoutNodeProvider::addLayoutConstraint` and `ScrollConstraint::addLayoutChild` without running layout solving, virtualization, Yoga updates, or `constrainChild`.
- `post_build_dependencies`: project C++ initialized `m_Drawables` as a static `drawable_order` list, including imported drawables, imported hidden flags, `ForegroundLayoutDrawable` reordering before its parent, inherited flattened draw-rule owner IDs, and layout `DrawableProxy` insertion. `sorted_drawable_order` now covers the graph-owned `sortDrawOrder()` phases: active draw-target grouping, before/after placement splicing, clipping proxy start/end interleaving, `clearRedundantOperations()` save-operation elision, and imported hidden drawable facts. Runtime `willDraw()` filtering, renderer commands, and GPU work remain out of scope.
- `post_build_dependencies`: project C++ initialized `m_DrawTargets` as static `draw_target_dependency_edges`, `draw_target_order`, and `draw_target_cycles`, including parent-ordered draw-rule groups, synthetic root target dependencies for resolved target drawables, and target-to-target dependencies from each target drawable's flattened rules. Active target linked-list grouping, placement splicing, clipping proxy interleaving, and save-operation flags are consumed by `sorted_drawable_order`; renderer commands and GPU work remain out of scope.
- `graph_validation`: aggregate structural diagnostics from already-projected
  graph facts: missing component parents, unresolved nonzero draw/clipping
  references, dependency cycles, dependency-node cycles, and draw-target cycles.
  These diagnostics do not mutate, reject, or execute the graph.

Verdict: the local-ID model now matches the first C++ graph expectations and the C++ probe output for the starter fixture set.

Implemented:

- `crates/rive-graph` as a pure projection over `rive-binary::RuntimeFile`.
- Artboard-local object slots matching C++ `Artboard::objects()`, preserving null/abstract slots, preserving C++ validation null slots such as invalid targeted constraints, text styles, nested animations, scroll-bar constraints, and feather effects, and excluding serialized records that C++ routes to other lists.
- File-level asset, view-model, and data-enum collection projections matching the C++ probe on reference fixtures, backed by `RuntimeFile::file_assets()`, `view_models()`, and `data_enums()` for C++ importer parity.
- Artboard animation and state-machine projections from `rive-binary` public helpers, including linear-animation keyed-object grouping and state-machine layer/input/listener/action/listener-input-type/data-bind grouping.
- C++ auto-generated empty state-machine projection for artboards without authored animations or state machines.
- Global-to-local relationship through `LocalObject { local_id, global_id }`.
- Component detection through generated schema inheritance.
- Capability flags for artboard, container, world transform, transform, and drawable categories.
- `on_added_dirty`-style parent resolution from local `parentId`.
- `on_added_dirty`-style draw graph relationships: `DrawTarget.drawableId` to nullable drawable local ID, `DrawRules.drawTargetId` to nullable active draw target local ID, and `ClippingShape.sourceId` to nullable source node local ID.
- `on_added_dirty`-style child indexing that mirrors C++ `ContainerComponent::children()`, preserving insertion order and duplicate child additions, including `LayoutComponent`/`Artboard` style adoption through `styleId`.
- `on_added_dirty`-style transform constraint registration lists that mirror
  C++ `Constraint::onAddedDirty -> TransformComponent::addConstraint` append
  order on each transform component.
- `on_added_clean`-style clipping shape projections: source shape locals from the clipping source subtree and clipped drawable locals from the clipping shape parent subtree.
- `on_added_clean`-style shape-paint container projections: `shape_paint_containers` mirrors C++ `ShapePaint::onAddedClean`, `ShapePaintContainer::addPaint`, `ShapePaintMutator::initPaintMutator`, `GradientStop::onAddedDirty`, and registered stroke-effect target links for exact C++ shape-paint containers. Paint mutation, effect execution, gradient stop sorting, path-effect application, renderer paint allocation, draw commands, and GPU work remain future runtime work.
- `on_added_dirty`-style `NSlicerDetails` projections: `n_slicer_details` mirrors C++ `NSlicerDetails::from`, ordered `addAxisX`/`addAxisY`, and patch-indexed `addTileMode` registration state for exact `NSlicer` and `NSlicedNode` details owners. NSlicer deformation math, patch solving, layout updates, path deformation, and render-path mutation remain future runtime work.
- `on_added_clean`-style shape render-path deformer projections: `shape_deformers` mirrors the cached first `RenderPathDeformer` ancestor that C++ `Shape::onAddedClean` discovers, currently exact `NSlicedNode` through `RenderPathDeformer::from`. Deformation math, `Path::buildPath` deformer application, gradient deformer updates, and point deformation remain future runtime work.
- `on_added_clean`-style skeletal projections: `skeletal_bones` mirrors `Bone::addChildBone` and `Bone::addPeerConstraint` import-time caches for exact `Bone` children and IK ancestor peer constraints, while `skeletal_skins` mirrors `Skin::onAddedDirty`, `Skinnable::from`, `Skinnable::skin`, and `Tendon::onAddedClean` registration order for exact `Mesh`/`PointsPath` skinnables. Bone transforms, IK solving, skin matrix allocation/update, vertex deformation, and skin/transform dirt propagation remain future runtime work.
- `on_added_clean`-style artboard host projections: `nested_artboards`, `component_lists`, and the combined `artboard_hosts` list mirror C++ `m_NestedArtboards`, `m_ComponentLists`, and `m_ArtboardHosts` for exact nested-artboard host variants and exact `ArtboardComponentList` objects. `ComponentListNode::map_rules` mirrors the static `ArtboardComponentList::addMapRule` table built by `ArtboardListMapRule::onAddedDirty`, including duplicate `viewModelId` overwrite behavior. Runtime host updates, nested-artboard cloning, component-list layout, list artboard instancing, data-context binding, virtualization, draw-index sorting, and advance scheduling remain out of scope.
- `on_added_clean`-style joystick projections: `joysticks` mirrors C++ `m_Joysticks`, `joysticks_apply_before_update` mirrors the artboard scheduling flag derived from `Joystick::canApplyBeforeUpdate`, and each `JoystickNode` records resolved handle source, x/y animation globals, and `Joystick::addDependents` nested remap targets in C++ y-then-x order. Joystick application, animation advancement, data-bind scheduling, and component update passes remain future runtime work.
- `on_added_clean`-style reset and advance projections: `resetting_components` mirrors C++ `m_Resettables` for exact nested-artboard host variants, `ArtboardComponentList`, and `CustomPropertyTrigger`, while `advancing_components` mirrors C++ `m_advancingComponents` for exact `Artboard`, nested-artboard host variants, layout/list/scroll/text/scripted objects admitted by `AdvancingComponent::from`. `Artboard::reset()`, `Artboard::advanceInternal()`, `advanceComponent()`, data-bind advancement, component updates, and frame scheduling remain future runtime work.
- `import`/`initialize`-style data-bind registrations: `ArtboardGraph::data_binds` mirrors artboard-owned `DataBindContainer` membership from C++ `DataBind::import`, `DataBindContainer::addDataBind`, and initialized `sortDataBinds()` order, while recording resolved target/converter IDs and types already proven by `rive-binary`; `StateMachineGraph::data_binds` now consumes `RuntimeFile::artboard_state_machine_graphs` so bindable-property target ownership follows the same C++ importer model as binary import. Data-context binding, dirty queues, property observers, converter execution, source/target mutation, state-machine execution, and data-bind advancement remain future runtime work.
- `import`-style state-machine scripted-object registrations: `StateMachineGraph::scripted_objects` mirrors C++ `StateMachineImporter::addScriptedObject`, `StateMachine::addScriptedObject`, and ordered `ScriptedObjectImporter::addInput` registrations for exact state-machine scripted-object owners. Script asset initialization, VM registration, cloning, script input hydration, script execution, and state-machine execution remain future runtime work.
- `build_dependencies`-style dependency nodes for real imported components plus synthetic path composers and text variation helpers. Every imported `Shape` gets one `PathComposerNode` and one synthetic `DependencyNode`, keyed by its shape local/global IDs and the registered path local/global IDs that C++ would make prerequisites of the path composer. Every imported `TextStyle` descendant with `TextStyleAxis` or `TextStyleFeature` children gets one synthetic text variation helper dependency node.
- `build_dependencies`-style component dependent lists for each imported
  component, mirroring the local-ID subset of C++ `Component::dependents()`
  after dependency construction and draw-target initialization. These lists are
  adjacency facts for later dirt scheduling; they do not run dirtying or frame
  updates.
- `on_added_dirty`-style mesh/path geometry projections: `meshes` mirrors C++ `MeshVertex::onAddedDirty`, `Mesh::addVertex`, and `Weight::onAddedDirty` for imported meshes, while `paths` mirrors `PathVertex::onAddedDirty`, `Path::addVertex`, and the same weight attachment behavior for imported paths. Vertex deformation, skinning math, `Path::buildPath`, contour/path tessellation, weight blending, and dirty propagation remain future runtime work.
- `build_dependencies`-style dependency edges for C++ parent dependency hooks, targeted constraints, IK constraints, IK chain off-branch children, draw targets, draw rules, clipping sources, skinning, Joystick custom-handle updates, scroll-constraint-to-scroll-bar links, scroll-constraint-to-layout-child links, clipping-shape path-composer links, follow-path target/parent links, text-follow-path target/text links, text variation helper links, stroke/fill/feather path-builder links, explicit `GroupEffect`/`ScriptedPathEffect` parent links, and linear/radial gradient paint-container links. `ParentChild` edges are deliberately narrower than structural `children`: they cover audited C++ parent dependency hooks such as `TransformComponent`, `Constraint`, `TextStyle`, `FocusData`, `SemanticData`, and `NSlicer`, while import-only graph records such as `DrawTarget`, `DrawRules`, `AxisX`, and `TextValueRun` retain hierarchy without dependency-order edges. Targeted-constraint edges now mirror C++ by making the constrained component depend on the target, rather than making the constraint object itself the dependent, and targeted constraints no longer receive the generic parent-child edge because the C++ override does not call `Super::buildDependencies`. `IKConstraint::buildDependencies` adds the extra target-to-constraint edge, and its `onAddedClean` IK-chain walk is modeled by making off-chain transform children of ancestor bones depend on the constrained tip bone. IK solving and transform dirt propagation remain future runtime work. `Skin::buildDependencies` is modeled by omitting the generic parent-child dependency for the skin child, making exact C++ skinnables (`Mesh` and `PointsPath`) depend on their skin, and making skins depend on tendon bones plus IK peer constraint parents discovered from `parentBoneCount`. `Joystick::buildDependencies` is modeled by omitting the generic parent-child dependency for all joysticks and adding parent-to-joystick plus handle-source-to-joystick edges only when `handleSourceId` resolves to an artboard-local `TransformComponent`. `ScrollBarConstraint::buildDependencies` is modeled by adding the resolved `ScrollConstraint` as a prerequisite while preserving the inherited parent dependency. `ScrollConstraint::buildDependencies` is modeled by making exact C++ `LayoutNodeProvider` children of the scroll content depend on the scroll constraint. `ClippingShape::buildDependencies` is modeled by making each clipping source shape's synthetic `PathComposer` dependency node a prerequisite of the clipping shape and by omitting the generic parent-child edge because the override does not call `Super::buildDependencies`. `FollowPathConstraint::buildDependencies` is modeled by making target shape/path composers prerequisites of the constraint, falling back to a direct path prerequisite for shape-less paths, and making the constrained parent depend on the constraint; `ListFollowPathConstraint` preserves those inherited edges. `TextFollowPathModifier::buildDependencies` is modeled by making target shapes' synthetic `PathComposer` nodes or direct target paths prerequisites of the modifier and making the owning `Text` depend on the modifier, without admitting text shaping or glyph layout. `TextVariationHelper::buildDependencies` is modeled by adding `artboard -> helper -> text` edges when imported text style axis/feature children cause C++ to allocate the helper; `TextVariationHelper::update()` and variable-font mutation remain future text runtime work. `Stroke::buildDependencies` is modeled by resolving C++ `ShapePaintContainer::pathBuilder()` as either a shape's synthetic `PathComposer` node or a real path-builder component. Effect-bearing `Fill::buildDependencies` reuses the same path-builder resolution only when registered stroke effects exist, and `Feather::buildDependencies` uses the owning paint container directly, with shapes again routed through their synthetic path composers. `LinearGradient::buildDependencies`, inherited by `RadialGradient`, is modeled by making gradients depend on the first owning `Node` above their shape paint, or the immediate paint container when no node exists; its `updateDeformer()` call remains future runtime/deformer state. Generic parent-child dependencies are deliberately skipped for the audited no-super shape/paint/effect families (`ClippingShape`, `Fill`, `Stroke`, `Feather`, `DashPath`, `TargetEffect`, `TrimPath`, `GroupEffect`, `ScriptedPathEffect`, `LinearGradient`, and `RadialGradient`) and for imported records that do not add C++ dependency edges so the graph does not overstate C++ build-dependency inheritance. None of these admit render-paint mutation, effect execution, IK solving, transform dirt propagation, deformer updates, text shaping, variable-font mutation, or draw-command behavior.
- `build_dependencies`-style list constraint registrations for exact `ListFollowPathConstraint` children whose parent is an exact `ArtboardComponentList`, matching C++ `ConstrainableList::from`, `ListConstraint::from`, and `ConstrainableList::addListConstraint`. This remains a static graph registration; `ArtboardComponentList::updateConstraints()`, `ListFollowPathConstraint::constrainList()`, list layout solving, and virtualization remain future runtime work.
- `build_dependencies`-style layout constraint registrations for exact `ScrollConstraint` content children accepted by `LayoutNodeProvider::from`, currently `LayoutComponent`, `NestedArtboardLayout`, and `ArtboardComponentList`, matching C++ `LayoutNodeProvider::addLayoutConstraint` and the reciprocal `ScrollConstraint::addLayoutChild` registration. This remains a static graph registration; layout solving, virtualization, Yoga updates, `ScrollConstraint::constrainChild()`, and scroll execution remain future runtime work.
- `build_dependencies`-style topological dependency order over explicit dependency-node edges, plus a filtered real-component local-ID order for existing callers, with cycle diagnostics for both node chains and all-real-component local-id chains. `ComponentNode::graph_order` is assigned from the C++ root-reachable `buildDependencies` order, including synthetic dependency-node slots such as `PathComposer`, while unreachable import-only components remain `None` instead of conflating them with C++'s default `graphOrder == 0`.
- `graph_validation`-style diagnostic aggregation through
  `ArtboardGraph::diagnostics`, reporting unresolved graph references and cycle
  summaries as static facts for later scheduler/runtime validation.
- Static drawable-order projection through `drawable_order`, matching C++ `Artboard::initialize()` for `m_Drawables.push_back(drawable)`, imported `DrawableFlag::Hidden` facts, `ForegroundLayoutDrawable` parent-bound reordering, parent-chain flattened draw-rule assignment, and layout `DrawableProxy` insertion. `sorted_drawable_order` then mirrors the graph-owned `sortDrawOrder()` passes by removing drawables with active flattened draw rules into target groups, splicing those groups before/after the resolved target drawable in `draw_target_order`, interleaving `ClipStartProxy`/`ClipEndProxy` nodes from static clipping-shape membership, applying `clearRedundantOperations()` save-operation elision, retaining imported hidden facts, and exposing the C++ `firstDrawable()->prevDrawable()` traversal order. Runtime `willDraw()` filtering, draw commands, and rendering remain future work.
- Static draw-target ordering through `draw_target_order`, matching C++ `Artboard::initialize()` for parent-ordered `DrawRules`, synthetic root target dependencies for resolved target drawables, flattened-rule target dependencies, and `DependencySorter` front-insertion ordering. `draw_target_dependency_edges` exposes root and flattened-rule edges, while `draw_target_cycles` preserves target-only cycle diagnostics for later draw-graph work. Unresolved `DrawTarget.drawableId` values remain in the nullable `draw_targets` projection instead of becoming target-order facts. Draw commands and rendering remain future work.
- `graph-inspect` JSON CLI.
- `tools/cpp-probe` C++ oracle plus `make cpp-compare`, which validates artboard object counts, local type keys, component names, serialized parent IDs, resolved parent local IDs, C++ child lists, C++ transform constraint lists, C++ component dependent lists, C++ graph-order values, component-list map rules, draw target/rule links, clipping relationships, artboard and state-machine data-bind registrations, state-machine scripted-object registrations, shape-paint container registrations, NSlicerDetails registrations, and audited shape deformer/skeletal/layout-constraint/data-bind/scripted-object/shape-paint/NSlicerDetails registration source behavior against C++.
